# Child Management Service — Architectural Vision

## Purpose

Child Management Service предназначен для управления пользователями (User) и их детьми (Child).  
Он обеспечивает безопасное хранение данных, соблюдение правил владения и доступа, а также инварианты целостности агрегатов.

Цель сервиса:
- изолировать бизнес-логику владения детьми
- централизовать управление правами доступа
- минимизировать распространение инфраструктурной логики в домене
- создать масштабируемый и тестируемый микросервис

---

## Business Value

Сервис обеспечивает:
- доверенную и безопасную модель владения детьми для пользователей
- возможность пользователя создавать свой User-агрегат и администратора управлять пользователями без нарушения инвариантов
- четкое разграничение прав доступа между ролями (Admin, User)
- архивирование детей через soft-delete с сохранением истории и возможности восстановления
- основу для расширений (истории детей, подписки, аналитика) без изменения доменной модели

---

## Scope

### In Scope

- управление пользователями (User)
- управление детьми пользователя (Child)
- соблюдение бизнес-инвариантов
- применение политик доступа
- администрирование пользователей

### Out of Scope

- аутентификация (логин, пароли, JWT)
- платежи и подписки
- уведомления и оповещения
- внешние интеграции и UI
- аналитика

---

## Core Concepts

- **User** — владелец детей, Aggregate Root
- **Child** — сущность, принадлежащая User
- **Admin** — системный актор с правами управления пользователями и их детьми
- **Aggregate Root** — точка входа для изменений в агрегате
- **Policy** — правила проверки доступа к операциям

---

## Design Principles

- **Domain-Driven Design (DDD)** — домен в центре, бизнес-правила внутри агрегатов
- **Clean Architecture** — слои с чёткой направленностью зависимостей
- **Explicit Policies** — права доступа отделены от сущностей
- **Infrastructure Agnostic Domain** — домен не знает про базу данных, HTTP или DI
- **Testable and Extensible** — возможность unit-тестов и будущих расширений без изменения модели

---

## High-Level Description

Все операции над детьми выполняются **через пользователя**.  
Child **не существует вне User**.  
Admin может управлять пользователями и их детьми, но **не владеет агрегатами напрямую**.  
Удаление Child в пользовательских сценариях реализуется как **soft-delete (archive)**.  
Права доступа проверяются через политики (Policy), а не в доменных сущностях.  
Сервис построен с учётом будущего расширения: истории, подписки, интеграции с другими сервисами.
