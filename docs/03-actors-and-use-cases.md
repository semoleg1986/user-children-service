# Child Management Service — Actors and Use Cases

## Purpose

Фиксирует все роли (Actors) и их действия (Use Cases) в системе.  
Служит **контрактом поведения**, чтобы:
- исключить недопонимания при разработке
- определить границы ответственности каждого актера
- формализовать политики доступа

---

## Actors

| Actor           | Description                           | Permissions / Scope                                                                                                                                                               |
|-----------------|---------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Admin**       | Системный администратор               | - Управление пользователями (создание, деактивация)<br>- Просмотр и управление детьми пользователей через User<br>- Архивация/восстановление Child<br>- Может видеть все истории детей |
| **User**        | Залогиненный пользователь (владелец детей) | - Создание своего User<br>- Управление своими детьми (создание, изменение, архивирование, восстановление)<br>- Просмотр своих детей<br>- Создание историй детей<br>- Не может управлять чужими детьми   |
| **AuthService** | Сервис аутентификации / идентификации | - Предоставляет идентификатор и роль актера<br>- Проверка токенов / авторизации                                                                                                   |

**Примечание:** В этой таблице **User** — это **актор** (роль того, кто выполняет Use Case). В доменной модели **User** — это **сущность** (Aggregate Root), объект с детьми. В Use Cases под «User» имеется в виду актор; при описании домена — сущность.

## Use Cases

### 1. Admin Use Cases

| Use Case        | Description                      | Preconditions     | Postconditions                                              |
|-----------------|----------------------------------|-------------------|-------------------------------------------------------------|
| Create User     | Создать нового пользователя      | Админ авторизован | Новый User добавлен                                         |
| Deactivate User | Деактивировать пользователя      | User существует   | User деактивирован, все Child остаются, доступ заблокирован |
| View Children   | Просмотр всех детей пользователя | User существует   | Admin получает список Child через User                      |
| Modify Child    | Изменение данных ребенка         | Child существует  | Child обновлен через User, инварианты соблюдены             |

| Use Case        | Description                     | Preconditions            | Postconditions                           |
|-----------------|---------------------------------|--------------------------|------------------------------------------|

## Rules and Constraints

1. Все операции над Child **должны проходить через User**.  
2. Admin **не владеет Child напрямую**, только через User.  
3. Пользователь **не может управлять чужими детьми**.  
4. Пользователь **может создавать только своего User**.  
5. Любая проверка доступа выполняется через **Policy**, а не в Entities.  
6. Создание, изменение и архивирование/восстановление Child **должно поддерживать инварианты User**.  
7. Все Use Cases должны быть покрыты unit / integration тестами.
8. Ошибки API возвращаются в формате RFC 7807 (см. `docs/07-error-format.md`).

## Notes

- Для каждой новой сущности (например, Story или Subscription) создается отдельный блок Use Cases.  
- Use Cases служат **основой для Application Layer**, где оркеструются действия через агрегаты и Policies.  
- Этот документ — **контракт для разработчиков, QA и архитектурного ревью**.
