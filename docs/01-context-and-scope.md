# Child Management Service — Bounded Context and Scope

## Context Name

**Child Management Context**

---

## Purpose of the Context

Контекст отвечает за управление пользователями и их детьми.  
Он инкапсулирует бизнес-логику владения, доступа и жизненного цикла Child и User,  
обеспечивая целостность агрегатов и соблюдение правил доступа.

---

## Responsibilities

Контекст обязан:

1. Управлять жизненным циклом **User** и их **Child**.
2. Поддерживать целостность агрегатов:
   - Child не существует вне User.
   - Все изменения Child происходят через User.
3. Обеспечивать корректные действия **Admin**:
   - просмотр и управление пользователями и их детьми.
4. Ограничивать действия **User**:
   - создавать только своего User
   - управлять только собственными детьми.
5. Проверять доступ через **Policies**:
   - Domain Entities не знают ролей.
6. Обеспечивать возможность расширений:
   - истории детей, подписки, интеграции, аналитика.

---

## Aggregate Structure

```shell
User (Aggregate Root)
└── Child
```
- **User** — точка входа для всех операций над Child.
- **Child** — сущность внутри User, не существует самостоятельно.


---

## External Dependencies

Контекст зависит от:

- **Auth Context**: предоставляет идентификатор и роль актёра (Admin/User).
- **Infrastructure Layer**: база данных, репозитории, Unit of Work.

Контекст не зависит от:

- реализации аутентификации (JWT, OAuth)
- HTTP / REST / GraphQL
- DI / контейнеров
- внешнего UI или клиентских приложений


## Integration Points

- **Входящие запросы**:
  - идентификатор пользователя
  - роль актёра (Admin / User)
  - операции: создать/удалить/просмотреть Child, управлять пользователем
- **Исходящие события** (потенциально):
  - доменные события о создании/удалении детей
  - уведомления в другие сервисы (опционально)

---

## Explicit Boundaries

В контексте запрещено:

- прямое создание или удаление Child вне User
- доступ User к чужим детям
- смешение ролей Admin и User в домене
- внедрение инфраструктурной логики в Entities
- хранение информации о ролях в агрегатах

---

## Rationale

- Чётко отделяет бизнес-логику от инфраструктуры.
- Упрощает масштабирование и тестирование.
- Позволяет легко расширять функционал без нарушения инвариантов.
- Поддерживает принципы DDD и Clean Architecture.
