# Child Management Service — Ubiquitous Language

## Purpose

Единый язык (Ubiquitous Language) фиксирует термины и определения,  
которые используются в домене, коде и коммуникации между командами.  
Это исключает неоднозначности и «домысловщину» при разработке.

---

## Core Terms

### User

- **Определение**: владелец детей (Child), Aggregate Root.
- **Роль в системе**: управляет своими детьми; подчиняется правилам доступа.
- **Взаимоотношения**:
  - владеет 0..* Child
- **Ограничения**:
  - не может управлять чужими детьми
  - все действия с детьми проходят через User
- **Синонимы**: Owner (не используется в коде, только для обсуждений)
- **Запрещено использовать**: Participant, Member (не отражает владение)

---

### Child

- **Определение**: сущность, принадлежащая User.
- **Роль в системе**: хранит персональные данные ребёнка, истории и другие связанные сущности.
- **Взаимоотношения**:
  - находится внутри User (композиция)
- **Ограничения**:
  - не существует без User
  - создаётся, изменяется, удаляется только через User
- **Синонимы**: Kid (разговорное), не используется в коде
- **Запрещено использовать**: Entity без User, отдельное хранение

---

### Admin

- **Определение**: системный актор с расширенными правами.
- **Роль в системе**:
  - управляет пользователями
  - управляет детьми пользователей через User
  - не владеет Child напрямую
- **Ограничения**:
  - действия всегда проходят через агрегаты User
  - не может менять правила агрегатов или инварианты
- **Синонимы**: System Operator
- **Запрещено использовать**: Superuser с прямым доступом к Child

---

### Aggregate Root

- **Определение**: сущность, являющаяся корнем агрегата, точкой входа для изменений.
- **Применение**: User — единственный Aggregate Root в данном контексте.
- **Ограничения**:
  - изменения Child только через методы User
  - гарантированная целостность данных

---

### Policy

- **Определение**: объект или набор правил, который определяет права доступа.
- **Применение**:
  - проверка возможности действия (add_child, remove_child, view_children)
- **Ограничения**:
  - Domain Entities не знают о ролях
  - Любая проверка роли/прав доступа происходит через Policy
- **Синонимы**: AccessRule, AccessPolicy
- **Запрещено использовать**: «if user.role == Admin» внутри Entity

---

### Repository

- **Определение**: интерфейс для работы с агрегатом (User).
- **Применение**:
  - get_by_id(user_id)
  - save(user)
- **Ограничения**:
  - работает только с агрегатом целиком
  - ChildRepository не используется
- **Синонимы**: Store, DAO (не используется в коде)
- **Запрещено использовать**: ORM внутри Domain, SQL в Entities

---

### Unit of Work

- **Определение**: объект, управляющий транзакциями при сохранении агрегатов.
- **Применение**:
  - commit
  - rollback
- **Ограничения**:
  - используется в Application слое
  - Domain не знает о Unit of Work

---

### Role

- **Определение**: тип пользователя (Admin, User)
- **Применение**:
  - передаётся в Application / Interface для проверки Policy
- **Ограничения**:
  - Domain Entities не хранят роль
  - Role не влияет на агрегаты напрямую

---

## Forbidden Terms

| Term                | Reason                                     |
|---------------------|--------------------------------------------|
| Participant         | не отражает владение детьми                |
| Member              | слишком общее                              |
| Superuser           | нарушает инварианты агрегата               |
| Kid                 | разговорное, не в коде                     |
| Entity (standalone) | Child не существует без User               |
| DB, SQL, ORM        | Domain должен быть infrastructure-agnostic |

---

## Notes

- Любое отклонение от зафиксированных терминов требует согласования с архитектором.
- Код, UML, тесты и документация должны использовать исключительно этот язык.
- Если появляется новая концепция (например, Story, Subscription), она фиксируется отдельным блоком в этом документе.
